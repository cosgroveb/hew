# hew ToT / Sub-Agent / Composition Analysis

## Context

hew is a minimal coding agent CLI (Go, stdlib only) with a two-tier API: Step() (single query-parse-execute cycle, no policy) and Run() (policy loop over Step). The core is an importable library. A previous attempt to add ToT to brainflower (C99 ECS agent) validated the concept but drowned in accidental complexity.

## Core Insight: ToT Is Not a Feature — It's a Pattern That Emerges From Two Primitives

The existing composability note says: "Agent is already a trajectory. NewAgent + Messages() + Step() compose into any multi-agent pattern without new abstractions." The one missing piece is trajectory I/O.

Two new primitives enable ToT, fan-out, and any multi-agent pattern via bash self-composition:

1. **Trajectory export** (`--dump-trajectory` / `--trajectory`): Serialize message history as JSON on exit
2. **Context seeding** (`--load-messages` + `AddMessages()`): Load prior messages to seed a child agent

With these, a parent hew instance can spawn child hew instances via its existing bash executor — ToT emerges as a system prompt, not as Go code.

## How ToT Works via Bash Self-Composition

A parent hew instance:
1. Researches the problem (normal Step/Run loop)
2. Asks LLM to generate N approaches
3. Creates N git worktrees via bash
4. Spawns `hew -p "implement approach X" --load-messages research.json --trajectory /path` in each worktree via its existing bash executor
5. Collects trajectories and diffs (cat files, git diff in worktrees)
6. Asks LLM to evaluate with anchored rubrics and forced ranking
7. Applies the winner's worktree

No new orchestration layer. No ToT-specific types or scoring interfaces in the library.

## Fan-Out and Sub-Agent Encapsulation

Fan-out without context encapsulation is just `xargs -P` — not an agentic pattern. The value comes from: divergent prompts (each child gets a different strategy), isolated state (worktrees), and convergent evaluation (compare, select).

ToT is the canonical fan-out pattern. Other fan-out patterns (task decomposition / map-reduce) compose from the same two primitives. No separate fan-out abstraction needed.

Messages-as-context (via AddMessages/--load-messages) is sufficient for sub-agent encapsulation. Children don't need separate system prompts, separate tool sets, or shared memory. The composition happens at the boundary (input seeding + output evaluation), not during execution.

## IPC: Why Regular Files Beat Everything Else

See hew-tot-ipc-analysis for the full breakdown. Summary:

- **Early termination**: Process groups via Setpgid + Kill(-pgid). Handled in Go executor, invisible to LLM.
- **Streaming observation**: Per-child JSONL event logs. Parent reads with cat/tail. No pipes, FIFOs, or sockets needed.
- **Result collection**: --trajectory flag + git diff in worktree.

The key constraint: the parent is an LLM issuing bash commands. Files are the universal bash interface. Everything else (pipes, FIFOs, UDS, shmem) requires setup ceremony that's fragile when generated by an LLM.

## Lessons From Brainflower

### What worked
- The ToT FSM (generate-execute-evaluate-select) works end-to-end
- The evaluator can differentiate implementation strategies with meaningful score spreads
- Git worktree isolation is sound
- The concept is validated — ToT picks better implementations than baseline for ambiguous problems

### What failed (and why it can't fail in hew)
- **eval_ruby bypass**: Compound tool hid decisions from the interception layer → hew has no compound tools, Step() returns the parsed action before execution
- **Segfault during concurrent sessions**: C99 memory management → Go eliminates this class of bug
- **ToT fires too early**: Triggered on research, not implementation → Start with manual triggering
- **Recursive ToT**: Branches spawned their own branches → Depth limit via flag/env var
- **Generator output parsing**: JSON embedded in prose → Prompt hardening, same issue but manageable

### Essential vs accidental complexity
- **Essential (exists in any ToT)**: When to branch, generating diverse approaches, isolating branch execution, evaluating results, applying winner, preventing recursion
- **Accidental (brainflower-specific)**: ECS lifecycle, approval tier interactions, compound tool abstraction, C99 memory, TUI rendering, Debian versioning

hew eliminates almost all accidental complexity. The essential concerns are real but tractable.

## Design Decisions

### "When to trigger" — manual first
Brainflower's confidence scorer never worked (always 0.0). LLM confidence is poorly calibrated for "is this problem hard enough to warrant parallel exploration." Manual triggering lets the human make the cost-benefit judgment. Semi-automatic heuristics can come later (agent explicitly considers alternatives, first attempt fails tests, task description contains explicit alternatives).

### Evaluation methodology — programmatic signals first, LLM-as-judge for tiebreaking
1. Tests pass (hard gate — failing = eliminated)
2. Lint clean (soft signal)
3. Diff size/complexity (programmatic proxy for minimality)
4. LLM-as-judge on the diff (anchored rubrics, forced ranking, evidence before scores)

The LLM evaluator only runs when there's a genuine choice between functionally correct solutions.

### Trajectory richness — StepResult accumulation
Run() currently discards StepResults. Accumulating []StepResult would provide step boundaries and execution metadata (cwd, exit codes) in the trajectory. More useful than raw messages for evaluation. Small internal change, not a design shift.

### Fork() vs AddMessages — no Fork needed
AddMessages + NewAgent is sufficient. The "full agent state" to preserve is just messages + cwd, and cwd is already a constructor parameter. Fork() would save two lines of code and add a method suggesting more complexity than exists.

### When multi-path exploration helps
From trajectory analysis:
- **Helps**: Clear diagnostic phase + ambiguous solutions, first attempt likely wrong in a way that wastes many steps, fix requires choosing between structurally different approaches
- **Noise**: Straightforward problems, bottleneck is understanding not implementation, narrow search space
- **Best predictor**: Agent's own visible uncertainty in reasoning ("I could do X or Y...")

## The 5 Implementation Features

| Feature | Scope | Est. Lines | Purpose |
|---------|-------|-----------|---------|
| `AddMessages([]Message)` | Library (agent.go) | ~5 | Seed agent with prior context |
| `Setpgid` on executor | Library (executor.go) | ~10 | Process group isolation for children |
| `--event-log /path` | CLI (main.go) | ~20 | Write JSONL events to file for streaming observation |
| `--trajectory /path` | CLI (main.go) | ~15 | Dump message history as JSON on exit |
| `--load-messages /path` | CLI (main.go) | ~20 | Seed conversation from JSON file |

Total: ~70 lines of Go. Everything else — the ToT pipeline, fan-out, evaluation, worktree management — is prompt engineering on the parent agent's system prompt.

## Implementation Order

1. Feature 4 (AddMessages) — dependency for Feature 3
2. Feature 5 (Setpgid) — independent, library scope
3. Feature 1 (--event-log) — CLI, independent
4. Feature 2 (--trajectory) — CLI, independent
5. Feature 3 (--load-messages) — CLI, depends on Feature 4

Library features first, CLI features second. Full plan at /tmp/hew-tot-plan.md.
