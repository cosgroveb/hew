# ToT Experiment: Next Steps — Claude-backed hew as orchestrator

## Goal

Test whether a Claude-backed `hu` instance can dynamically generate divergent
approaches and orchestrate ToT end-to-end, replacing the hardcoded orchestration
that Claude Code performed in the initial experiment.

## Background

The initial ToT experiment (2026-03-16) showed that ToT-Split improves
GPT-OSS-120B outcomes from ~50% per-attempt to 80-100% via best-of-3 selection.
But the orchestration was done by Claude Code (the CLI agent running the
experiment), not by `hu` itself. The approaches were hardcoded in a Python
script, not generated by a model.

The question: can a Claude-backed `hu` instance do the full ToT loop?
1. Research the problem
2. Generate 3 distinct implementation strategies
3. Create work directories with divergent APPROACH.md files
4. Spawn child `hu -p` processes (GPT-OSS) to implement each branch
5. Collect test results from each branch
6. Select the winner
7. Present the result

This is a harder test than RLM because it requires meta-reasoning
(generating diverse strategies) in addition to orchestration (dispatching
and collecting).

## What we already know

From the RLM experiment:
- Claude Opus follows decomposition instructions faithfully (F1=0.99 on RLM)
- GPT-OSS-120B ignores orchestration instructions entirely
- The winning RLM prompt is in ~/AGENTS.md (v3 with scoped bulk ops)

From the ToT experiment:
- Best-of-3 selection turns 50% branch success into 80-100% system success
- Approach-specific instructions (divergent APPROACH.md) outperform vague prompts
- Test pass count is sufficient for winner selection (no LLM-as-judge needed)
- Complex tasks (refactoring) benefit most from ToT

## The ToT prompt

The ToT orchestration prompt needs to be added to ~/AGENTS.md (via chezmoi).
It should teach the parent hu instance to:

1. Read the task and assess complexity
2. If the task is ambiguous or has multiple valid approaches, generate
   3 distinct strategies (not cosmetic variations — structurally different)
3. For each strategy, create a temp directory with the fixture + APPROACH.md
4. Spawn child `hu -p` processes to implement each strategy:
   ```bash
   hu -p "Read APPROACH.md and README.md. Implement the solution using the
   specified approach. Run make test to verify. Only modify allowed files." \
     --trajectory /tmp/tot-branch-N.json --max-steps 30
   ```
5. After all children complete, check test results:
   ```bash
   for f in /tmp/tot-branch-*.json; do
     cd <branch-work-dir>
     go test ./... -v -count=1 -race 2>&1 | tail -5
   done
   ```
6. Select the branch with the most passing tests
7. Apply the winner (copy files back or report the diff)

**Key constraints for the prompt:**
- Children must be GPT-OSS (configure via env vars in the spawn command):
  ```bash
  HEW_BASE_URL="http://localhost:8000/v1" HEW_API_KEY="not-needed" \
    HEW_MODEL="gpt-oss-120b" hu -p "..." --trajectory ... --max-steps 30
  ```
  Wait — on the work Mac there's no local GPT-OSS. Children would also use
  Claude via Anthropic API. That's fine for testing the orchestration pattern.
  To test with GPT-OSS children, run from spark instead.
- Dispatch sequentially (max 3 children, no backgrounding)
- Approach generation must be specific and structural, not vague

## Eval methodology

Use the same 3 task fixtures from the initial experiment:
- `fixtures/task1-ratelimiter/` — rate limiter implementation (13 tests)
- `fixtures/task2-refactor/` — god object refactoring (15 tests)
- `fixtures/task3-multibug/` — interrelated bug fixes (8 tests)

### Configurations

| Config | Parent | Children | What it tests |
|--------|--------|----------|---------------|
| Baseline | Claude hu | N/A | Single-path Claude performance |
| ToT-Dynamic | Claude hu | Claude hu | Can Claude orchestrate + implement ToT? |
| ToT-Split-CG | Claude hu | GPT-OSS hu | Claude orchestrates, GPT-OSS implements (from spark) |

### Runs

5 runs per configuration per task. Compare:
- **Pass rate**: fraction of runs where all tests pass
- **Best-of-N lift**: does ToT-Dynamic beat Baseline?
- **Strategy diversity**: are the generated approaches genuinely different?
- **Orchestration fidelity**: does hu follow the ToT prompt (dispatch children,
  collect results, select winner)?

### Scoring

Same as initial experiment: `go test ./... -v -count=1 -race`, count
passed/failed, select winner by test count.

For strategy diversity, manual review: read the generated APPROACH.md files
and classify as "structurally different" or "cosmetic variation."

## Running the eval

```bash
# Clone fixtures to a temp dir
EVAL_DIR=$(mktemp -d /tmp/tot-eval-XXXXXX)
cp -r ~/code/hew/docs/tot-experiment/fixtures/* "$EVAL_DIR/"

# Run baseline (single Claude hu, no ToT)
cd "$EVAL_DIR/task1-ratelimiter"
TASK_PROMPT=$(cat README.md)
hu -p "$TASK_PROMPT" \
  --trajectory /tmp/baseline-task1.json \
  --max-steps 30

# Run ToT-Dynamic (Claude hu orchestrates and spawns children)
# The ToT prompt in ~/AGENTS.md tells hu to generate approaches and dispatch
cd "$EVAL_DIR/task1-ratelimiter"
hu -p "Implement the rate limiter described in README.md. Use Tree of Thought:
generate 3 structurally different approaches, implement each in a separate
work directory using child hu processes, test each, and select the best." \
  --trajectory /tmp/tot-dynamic-task1.json \
  --max-steps 50

# Score
go test ./... -v -count=1 -race
```

## Expected outcomes

**Baseline (Claude single-path)**: Should pass most tasks easily. Claude is
a strong model. The interesting question is whether it ever picks a suboptimal
approach that ToT would have avoided.

**ToT-Dynamic (Claude orchestrates Claude)**: Should work — Claude follows
instructions well. The question is whether the overhead of generating
approaches + spawning children is worth it compared to just solving directly.
For simple tasks, probably not. For ambiguous tasks, maybe.

**ToT-Split-CG (Claude orchestrates GPT-OSS)**: This is the replication of
the initial experiment but with Claude generating approaches dynamically
instead of hardcoding them. Should match or exceed the initial experiment's
80-100% pass rates.

## Decision tree

- **Baseline passes all tasks**: ToT may not help for Claude-class models.
  The value is for weaker models (GPT-OSS). Focus on ToT-Split-CG.
- **ToT-Dynamic beats Baseline**: ToT adds value even for strong models.
  Investigate which tasks benefit and why.
- **ToT-Dynamic matches Baseline**: The approach generation is working but
  the model doesn't need help. ToT is an insurance policy, not a boost.
- **Orchestration fails**: hu doesn't follow the ToT prompt. Debug using
  `--dump-system-prompt` and trajectory inspection. May need prompt iteration
  similar to RLM experiment.
