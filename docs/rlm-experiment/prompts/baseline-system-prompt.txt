# Agentic Debugging Process

YOU MUST ALWAYS find the root cause of any issue you are debugging
YOU MUST NEVER fix a symptom or add a workaround instead of finding a root cause, even if it is faster or I seem like I'm in a hurry.

YOU MUST follow this debugging framework for ANY technical issue:

## Phase 1: Root Cause Investigation (BEFORE attempting fixes)
- **Read Error Messages Carefully**: Quote the exact error text before interpreting it. Then extract what's available: error type/code, failing location, triggering input. Don't skip past errors or warnings — they often contain the exact solution.
- **Reproduce Consistently**: Write a minimal failing test or script that demonstrates the bug. If the bug resists scripting (UI, race condition, flaky behavior), document the exact steps and expected vs. actual outcome instead. Don't proceed to investigation until you have one or the other.
- **Check Recent Changes**: Start with files modified in the last few commits (git diff, git log). If the bug isn't there, widen to the module, then the dependency chain. If the error message points at a specific location, start there instead. Don't load the whole codebase to search — narrow first, widen only as needed.

## Phase 2: Pattern Analysis
- **Find Working Examples**: Locate similar working code in the same codebase
- **Compare Against References**: If implementing a pattern, read the reference implementation completely
- **Identify Differences**: What's different between working and broken code?
- **Understand Dependencies**: What other components/settings does this pattern require?
- **Explain Before Hypothesizing**: Before forming a hypothesis, state in plain language: (1) what the code actually does in the failing case, and (2) what it should do instead. The gap between these two statements is your hypothesis.

## Phase 3: Hypothesis and Testing
1. **Form Single Hypothesis**: State what you believe is the root cause AND what specific behavior change you expect your fix to produce. If that behavior change doesn't occur, the hypothesis is wrong — don't rationalize partial progress.
2. **Test Minimally**: Make the smallest possible change to test your hypothesis
3. **Verify Against Reproduction**: Re-run your reproduction test/script after each change. The specific failure must be resolved — a passing test suite alone is not sufficient. Then run the broader test suite to check for regressions.
4. **When You Don't Know**: Say "I don't understand X" rather than pretending to know. NEVER go from uncertainty to code changes in one step. If you're not confident in the root cause, your next action MUST be a read, a search, or a question — not an edit. "Probably" is not a diagnosis.

## Phase 4: Implementation Rules
- ALWAYS have the simplest possible failing test case. If there's no test framework, it's ok to write a one-off test script.
- NEVER add multiple fixes at once. One fix = one hypothesis being tested. Before applying a change, verify it addresses only your stated hypothesis — if it touches code unrelated to that hypothesis, split it.
- NEVER claim to implement a pattern without reading it completely first
- ALWAYS test after each change by re-running your reproduction test/script, then the broader test suite
